# API: добавление, настройка и удаление устройств (schema-driven)

Дополняет `BACKEND_API.md`. Здесь — именно та часть API, что нужна компонентам
`RoomDevicesManager` / `DeviceConfigSheet` / `AddDeviceSheet`
(см. `DeviceConfigurator.jsx`).

## 1. Идея

У каждого типа устройства — своя схема параметров, а не жёстко прибитая форма.
Бэкенд отдаёт схему один раз на тип, фронтенд рендерит форму по ней. Если у типа
нет параметров — схема пустая, и это нормальное, ожидаемое состояние, а не ошибка.

```
GET /api/device-types            → список схем на все типы
GET /api/device-types/:type      → схема одного типа
```

### Пример ответа `GET /api/device-types/climate`

```json
{
  "type": "climate",
  "label": "Кондиционер",
  "params": [
    { "key": "mode", "label": "Режим", "control": "select",
      "options": ["cool", "heat", "fan", "off"], "default": "cool" },
    { "key": "fanSpeed", "label": "Скорость вентилятора", "control": "select",
      "options": ["авто", "низкая", "средняя", "высокая"], "default": "авто" },
    { "key": "swing", "label": "Поворотные жалюзи", "control": "toggle", "default": false }
  ]
}
```

### Пример ответа `GET /api/device-types/window_sensor`

```json
{ "type": "window_sensor", "label": "Датчик окна", "params": [] }
```

Пустой `params` — сигнал фронтенду показать пустое состояние в
`DeviceConfigSheet`, а не пытаться отрисовать форму из ничего.

## 2. Поддерживаемые типы контролов (`control`)

| control  | поля схемы                          | что рендерит фронтенд |
|----------|--------------------------------------|------------------------|
| `toggle` | `default`                             | переключатель вкл/выкл |
| `slider` | `min`, `max`, `step`, `unit`, `default` | ползунок с текущим значением |
| `select` | `options[]`, `default`                | выпадающий список |
| `number` | `min`, `max`, `step`, `unit`, `default` | числовое поле |

Новый тип устройства = новая запись в конфиге схем на бэкенде. Фронтенд ничего
не меняет — `ParamField` уже умеет все четыре контрола.

## 3. Устройство: чтение и запись параметров

```
GET   /api/devices/:id
```
```json
{
  "id": "dev_42",
  "roomId": "r2",
  "type": "climate",
  "name": "Кондиционер",
  "params": { "mode": "cool", "fanSpeed": "авто", "swing": false },
  "state": { "targetTemp": 22, "currentTemp": 23.4 },
  "diagnostics": { "battery": null, "linkquality": 89 }
}
```

`params` — то, что настраивается через `DeviceConfigSheet` (схема). `state` —
оперативные показания/управление (то, что меняется через тумблер/слайдер на
главном экране, не через настройки). Разделены намеренно: изменение `params`
идёт через отдельный, более «тяжёлый» эндпоинт с валидацией по схеме, а
`state` — через частые лёгкие вызовы toggle/slider.

```
PATCH /api/devices/:id/params      { "fanSpeed": "высокая" }
```
Сервер валидирует каждое поле против схемы типа устройства (`select` — значение
должно быть из `options`, `slider`/`number` — в диапазоне `min..max`). При
нарушении — `422` с списком невалидных полей, форма на фронтенде подсвечивает
именно их.

```json
// 422
{ "errors": [{ "key": "fanSpeed", "reason": "not_in_options", "options": ["авто","низкая","средняя","высокая"] }] }
```

## 4. Добавление устройства в комнату

```
POST /api/rooms/:roomId/devices
{ "type": "light", "name": "Бра над зеркалом" }
```

Сервер подставляет `params` по умолчанию из схемы типа (`defaultParamsFor`),
создаёт запись, возвращает её:

```json
{ "id": "dev_77", "roomId": "r3", "type": "light", "name": "Бра над зеркалом",
  "params": { "colorTemp": 3500, "powerOnBehavior": "Восстановить состояние" } }
```

Если устройство подключается физически (Zigbee) — этот вызов делается после
подтверждения найденного устройства в discovery-потоке
(`POST /api/discovery/:ieee/confirm`, см. `BACKEND_API.md`); поле `ieeeAddress`
дополнительно прикрепляется к записи.

## 5. Убрать из комнаты vs удалить совсем

Два разных действия, обе кнопки есть в `DeviceConfigSheet`:

```
PATCH /api/devices/:id            { "roomId": null }   # убрать из комнаты,
                                                        # устройство остаётся в системе
                                                        # («неприписанное», видно в общем списке)
DELETE /api/devices/:id                                 # удалить совсем: запись,
                                                         # история показаний, привязка к Zigbee
```

`PATCH roomId: null` — обратимое действие (устройство можно назначить в другую
комнату позже без переобнаружения). `DELETE` — необратимое, поэтому на фронтенде
оформлено отдельной, визуально «опасной» кнопкой и должно требовать
подтверждения в диалоге, а не быть доступным одним кликом.

## 6. Валидация на границе

- Схема — источник истины и на бэкенде (валидация `PATCH .../params`), и на
  фронтенде (какие контролы рисовать) — одна и та же схема, не две копии.
- Если у типа `params: []`, `PATCH /api/devices/:id/params` для него всегда
  возвращает `400 nothing_to_configure` — фронтенд просто не показывает форму,
  но и бэкенд не должен молча принимать мусорные поля.
