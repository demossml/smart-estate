# AI-агенты в «Умной усадьбе»: архитектура на протоколе Hermes

## 1. Зачем Hermes

Hermes — открытый MQTT-протокол голосового ассистента (изначально Snips.AI, сейчас
поддерживается Rhasspy, openHAB и др.). Он описывает событийную цепочку:

```
hotword → ASR → NLU → dialogue manager → TTS / intent handler
```

каждый шаг — это публикация/подписка на MQTT-топики вида `hermes/asr/textCaptured`,
`hermes/nlu/intentParsed`, `hermes/tts/say` и т.д.

Мы уже используем MQTT-брокер для Zigbee2MQTT (`device_announce`, `permit_join` —
см. раздел discovery в документации устройств). Hermes ложится на тот же брокер,
поэтому голос, детерминированные интенты и AI-агент оказываются в одном событийном
потоке, с одним логом — а не тремя разными интеграциями.

## 2. Главный принцип: AI — не по умолчанию, а по необходимости

В 2025 году Amazon (Alexa+) и Google (Gemini for Home) заменили детерминированное
распознавание команд генеративными моделями почти везде — и получили обогреватель,
включавшийся на 86°F по ночам, будильники, которые нельзя было отменить, и ответы
про погоду песней Pitbull. Home Assistant в это же время остался в основном на
детерминированных правилах и получил больше доверия пользователей.

**Вывод, который мы закладываем в архитектуру:** быстрый детерминированный путь
(grammar-based NLU) обрабатывает команду первым и бесплатно. AI-агент вызывается
только тогда, когда:

- фразу нельзя разложить на `устройство + действие` по фиксированной грамматике;
- нужно сопоставить несколько сигналов, чтобы понять причину/конфликт;
- нужно превратить свободный текст в структурированный сценарий;
- нужна суммаризация большого объёма телеметрии в текст.

Если задачу можно решить if/else или regexp — она остаётся в NLU-грамматике или в
движке сценариев, без единого обращения к LLM.

### Таблица разделения ответственности

| Не требует AI (grammar/rule engine) | Требует AI-агента (function calling) |
|---|---|
| «Включи свет в гостиной» — устройство и глагол есть явно | «Мне холодно» — нет ни устройства, ни точного действия |
| IF-THEN сценарии (раздел «Сценарии») | Свободное описание сценария в 2–3 предложениях с оговорками |
| Чтение значения датчика | «Почему в доме жарко?» — нужно сопоставить климат/окна/время |
| Таймеры и расписания | Конфликт двух активных сценариев |
| Открыть/закрыть конкретное устройство по имени | Диалог с гостем через интерком у ворот |
| — | Сводка/дайджест за день или неделю |
| — | Проактивные подсказки по паттернам использования |

## 3. Топология MQTT-топиков

```
hermes/hotword/<site>/detected          # локально, офлайн, без AI
hermes/asr/<site>/textCaptured          # текст команды
hermes/nlu/intentParsed                 # совпало с grammar — интент выполняется напрямую
hermes/nlu/intentNotRecognized          # НЕ совпало → триггер для AI

smartestate/ai/query                    # запрос к AI-агенту: {text, context}
smartestate/ai/response                 # структурированный ответ агента
smartestate/ai/pendingAction            # предложенное, но не выполненное действие
smartestate/ai/suggestion               # проактивная подсказка по паттерну

hermes/tts/say                          # финальный ответ пользователю голосом
```

`smartestate/ai/*` — наше расширение поверх Hermes по тому же принципу именования,
чтобы агентские события были видны в том же MQTT-логе, что и голосовые/device-топики.

## 4. Поток запроса

1. Микрофон-сателлит: `hermes/hotword/detected` → `hermes/asr/textCaptured`.
2. Детерминированный NLU (фиксированные grammar-фразы, как в Rhasspy `sentences.ini`)
   пытается сматчить `текст → интент(устройство, действие)`.
   - Совпало → выполняется напрямую через существующий MQTT-топик устройства
     (Zigbee2MQTT `set`), публикуется `hermes/tts/say` с подтверждением. **AI не
     вызывается.**
   - Не совпало → `hermes/nlu/intentNotRecognized` → публикуется `smartestate/ai/query`
     с текстом и срезом контекста (см. ниже).
3. Backend вызывает выбранного в разделе «Управление» провайдера (Anthropic /
   OpenAI / OpenRouter / Ollama) с function calling.
4. Ответ агента — это либо прямой текст (TTS), либо вызов одного из разрешённых
   инструментов (см. раздел 6). Необратимые действия не выполняются немедленно —
   публикуются как `smartestate/ai/pendingAction` и ждут подтверждения в приложении
   или голосом («да»/«подтверждаю»).

### Контекст, который передаётся агенту

```json
{
  "text": "в гостиной холодно",
  "rooms": [{ "id": "r2", "name": "Гостиная", "devices": ["light:Люстра:on", "climate:Кондиционер:on:22°"] }],
  "activeScenarios": ["Открыто окно и включён кондиционер → Выключить кондиционер"],
  "recentEvents": ["09:12 окно эркер открыто", "09:40 присутствие: гостиная"]
}
```

Контекст обрезан до текущей комнаты + активных сценариев + последних событий —
не весь дом целиком, чтобы не раздувать промпт и не тратить лишние токены.

## 5. Гвардрейлы (обязательные)

- **Подтверждение для необратимых действий**: ворота, замки, климат за пределами
  16–28°, удаление сценария — всегда через `pendingAction`, никогда не выполняются
  агентом напрямую.
- **Токен провайдера не покидает бэкенд** — фронтенд получает только замаскированное
  значение после сохранения (см. `BACKEND_API.md`, раздел AI providers).
- **Один агентский вызов = один запрос**, без фоновых циклов «пока не добьётся
  своего» — снижает риск зацикливания и неконтролируемых трат токенов.
- **Rate limit** на `smartestate/ai/query`: не более N запросов в минуту на дом,
  чтобы случайный шум/тестовые фразы не тратили бюджет провайдера.
- **Логирование** каждого агентского решения (запрос → инструмент → результат)
  для аудита — то же самое, чего не хватало в первых версиях Alexa+/Gemini for Home
  и что стало причиной публичных нареканий на непредсказуемость.

## 6. Инструменты (function calling)

Агенту доступен ограниченный набор функций, а не произвольный доступ к системе:

| Функция | Действие | Подтверждение |
|---|---|---|
| `get_room_state(roomId)` | Чтение состояния комнаты | не требуется |
| `set_device_state(deviceId, state)` | Включить/выключить/установить яркость | требуется, если устройство — климат/ворота/замок |
| `adjust_climate(deviceId, delta)` | Изменить целевую температуру | требуется, если |Δ| > 2° |
| `create_scenario(condition, action)` | Добавить IF-THEN сценарий | требуется |
| `summarize(period)` | Сформировать текстовую сводку | не требуется |

## 7. Что уже реализовано в интерфейсе

Раздел **«Управление» → «Голосовое управление (Hermes)»**:

- визуальный пайплайн `Hotword → ASR → NLU → AI-агент`;
- тестовое поле для эмуляции `hermes/asr/textCaptured` — показывает, ушла ли фраза
  по детерминированному пути или к агенту, с объяснением почему;
- очередь **«Ждут подтверждения»** — pending actions с кнопками Да/Отклонить;
- **«Предложения по паттернам»** — проактивные подсказки, превращаются в сценарий
  одним нажатием;
- **«Сводка за сегодня»** — генерируется из текущей телеметрии (энергия, открытые
  окна, свет, число активных автоматизаций).

## 8. Что дальше (не реализовано в текущей версии)

- Реальное подключение ASR/TTS (Whisper.cpp/Vosk локально — приоритет приватности,
  по примеру Home Assistant, а не облачная отправка аудио, как у Alexa+ после
  марта 2025).
- Аппаратные сателлиты (ESP32 + микрофон/динамик) на Hermes MQTT.
- Реальный вызов LLM API вместо эмулированной классификации фраз в демо.
- Fine-tuning grammar-интентов на основе тех фраз, которые чаще всего улетают к AI —
  чтобы со временем детерминированный путь покрывал больше и агент вызывался реже.
