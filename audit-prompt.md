# Промпт для аудита Smart Estate (Умная Усадьба)

## Описание приложения

Smart Estate — это система умного дома на базе Zigbee2MQTT + собственный Node.js бэкенд + React PWA фронтенд.

### Компоненты

```
[Zigbee2MQTT] ──MQTT──→ [Backend :8788] ──REST API──→ [Frontend :5173 / PWA]
                              │
                              ↓
                        [SQLite: smart-estate.db]
```

### Backend
- **Express.js** на Node.js, порт 8788
- SQLite через **better-sqlite3** (синхронный)
- Подписка на **MQTT** (zigbee2mqtt/#) — приём телеметрии с Zigbee-устройств
- REST API для фронта: устройства, телеметрия, комнаты, сценарии, события
- Есть демо-режим (эмуляция устройств, если MQTT недоступен)
- WebSocket для real-time обновлений
- Запуск: `node dist/index.js` (скомпилированный TypeScript)

### Frontend
- **React 19** + **TypeScript** + **Vite** + **Tailwind CSS 4**
- PWA (vite-plugin-pwa)
- Компоненты: lucide-react + react-router-dom
- Страницы: Dashboard, Devices, Rooms, Climate, Energy, Gates, Scenarios, Events, Analytics
- WebSocket подключение для real-time данных (/ws)
- Анимации и микровзаимодействия

### Инфраструктура
- Docker: **zigbee2mqtt**, **mosquitto** (MQTT брокер)
- **systemd** сервис: smart-estate.service
- Пути: `/home/admingimolost/smart-estate/`

### Текущие известные проблемы

1. **Дублирование устройств в БД**
   - Z2M публикует телеметрию некоторых датчиков без `ieee_address` в payload
   - Код использует `data.ieee_address || data.ieeeAddr || friendlyName` — если нет IEEE, берёт имя
   - Создаются записи с `ieee_addr = "Датчик качества воздуха"` (текст вместо MAC)

2. **Фиктивные данные датчиков**
   - Датчик HOBEIAN ZG-204ZK (24 ГГц радар присутствия) НЕ имеет сенсоров CO₂/VOC/формальдегида
   - Z2M эмулирует эти поля на основе прошивки
   - Temperature/Humidity тоже эмулируются (нет физических сенсоров)
   - На фронте CO₂/VOC показываются с единицами ppm/ppb/mg/m³ — это вводит в заблуждение

3. **Нестабильная телеметрия**
   - Датчик на батарейках (шлёт только при изменении состояния presence)
   - Temperature/humidity не обновляются часами
   - Последние данные могут быть устаревшими на 1-2 часа

4. **Отсутствие тестов**
   - В package.json есть vitest и supertest, но нет тестовых файлов
   - Нет автоматической проверки регрессий

5. **Проблемы с инициализацией БД**
   - При первом запуске БД создаётся, но таблицы могут не создаться если модуль db.ts не загружен корректно
   - Зависит от порядка: schema создаётся при импорте db.ts

6. **Демо-режим не синхронизирован с реальной схемой**
   - demo.ts генерирует тестовые данные, но может не совпадать с реальной схемой после изменений

## Задачи для аудита

### Задача 1: Полный аудит кода backend

Проверьте каждый файл в `/home/admingimolost/smart-estate/server/src/` на:

**Безопасность**:
- SQL-инъекции (используются ли параметризованные запросы?)
- Защита от XSS/CSRF (csrfProtect на всех мутирующих endpoints?)
- Валидация входных данных (Zod схемы используются везде?)
- Rate limiting на критических endpoints
- API ключи, авторизация (проверить что приватные эндпоинты защищены)

**Надёжность**:
- Обработка ошибок (try/catch везде?)
- Graceful shutdown (закрытие MQTT, SQLite при SIGTERM)
- Переподключение MQTT при обрыве
- Deadlock-ситуации с SQLite (sync better-sqlite3 + асинхронные запросы)
- Memory leak: утечки через query()-замыкания, Map без очистки

**Производительность**:
- N+1 запросы в API
- Индексы в SQLite (проверить CREATE INDEX)
- Большие ответы (>1MB) без пагинации
- Блокировка event loop синхронными SQL-запросами

**Архитектура**:
- Всё в одном файле api.ts (60+ роутов) — нужно разделение?
- Нет слоёв: контроллеры → сервисы → репозитории
- Дублирование логики (поиск устройств по имени повторяется в 3+ местах)

**Логи**:
- Логирование ошибок с контекстом (device_ieee, endpoint, trace)
- Нет ли утечки чувствительных данных в логи (токены, пароли)
