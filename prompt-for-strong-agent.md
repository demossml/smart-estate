## Архитектура приложения

### Стек
- **Frontend**: Svelte 5 + Vite + TypeScript + PWA
- **Backend**: Hono.js (Bun runtime) — REST API на порту 8788
- **База**: SQLite (`data/evotor.db`), инициализация через `init.sql`
- **MQTT**: Zigbee2MQTT на порту 1883 (топик `zigbee2mqtt/+/`)
- **Docker**: 3 контейнера — app, caddy (обратный прокси), zigbee2mqtt
- **Деплой**: `/home/admingimolost/smart-estate/`

### Компоненты

**Backend:** `packages/backend/src/`
- `index.ts` — точка входа, запуск Hono, WebSocket, MQTT
- `routes/` — API эндпоинты (devices, sensors, dashboard, etc.)
- `mqtt/` — MQTT-обработчики (телеметрия, bridge, регистрация)
- `db/` — SQLite инициализация, миграции
- `models/` — типы данных (Device, Telemetry, SensorType)
- `services/` — бизнес-логика (device, telemetry, status, dups)
- `scripts/` — вспомогательные скрипты

**Frontend:** `packages/frontend/src/`
- `routes/` — SvelteKit-страницы (dashboard, sensors, energy...)
- `components/` — UI компоненты (DeviceTile, ScenariosTab, etc.)
- `components/ui/` — дублирующие UI компоненты
- `lib/` — утилиты, API-клиент
- `app.html`, `layout.svelte` — корневые шаблоны

**Данные:**
- Сенсоры TS0601 (Tuya) — температура, влажность, CO₂, VOC, формальдегид, освещённость
- MQTT телеметрия — реальные данные с датчиков
- SQLite — устройства, телеметрия, конфигурация

### Поток данных
1. Zigbee2MQTT получает данные с датчиков → публикует в MQTT топик `zigbee2mqtt/Датчик`
2. Backend подписан на `zigbee2mqtt/+` → получает телеметрию
3. Обработчик MQTT сохраняет в SQLite по `device_id` (friendly_name из топика)
4. API отдаёт данные фронтенду
5. WebSocket обновляет в реальном времени

## Известные проблемы (из аудита)

### 1. Dashboard падает с SQL syntax error [CRITICAL]
Скрипт `audit_backend.py` показал: `GET /api/dashboard` возвращает HTTP 500. Ошибка — не экранированная строка `'1 minute'` в SQLite запросе. Dashboard — главная страница приложения, без неё весь смысл теряется.

**Где смотреть:** `routes/dashboard.ts` — SQL конструкция с `'1 minute'` без параметризации.

### 2. Телеметрия не пишется — фильтр по ieee_address [CRITICAL]
Zigbee2MQTT публикует данные без поля `ieee_address` (только `friendly_name`). В `mqtt/handler.ts` стоит фильтр: `if (!payload.ieee_address) return`. Из-за этого **вся телеметрия молча дропается**.

Более ранние версии кода использовали поиск `ieee_address` по `friendly_name` в кэше или БД — этот механизм был удалён/заменён.

**Где смотреть:** `mqtt/handler.ts`, сравнить со старой логикой записи телеметрии.

### 3. Дубли компонентов на фронтенде (4 пары) [HIGH]
В `components/` и `components/ui/` лежат одинаковые компоненты:
- `ScenariosTab.svelte` — в обоих местах
- `AddDeviceModal.svelte`
- `DeviceTile.svelte`
- `EnergyTab.svelte`

При этом в разных файлах они импортируются из разных путей. Могут быть мелкие расхождения.

**Где смотреть:** `frontend/src/components/` и `frontend/src/components/ui/`

### 4. PWA устаревший [MEDIUM]
- `manifest.webmanifest` иконки ведут на несуществующие файлы (`/icon-192x192.png`)
- Service Worker (`sw.js`) не загружается при тесте
- В SvelteKit PWA подключается через `vite-plugin-pwa`, но конфиг не настроен

**Где смотреть:** `frontend/manifest.webmanifest`, `frontend/static/sw.js`

### 5. Кэш MQTT-устройств — потенциальная утечка памяти [MEDIUM]
В `mqtt/handler.ts` есть `deviceCache = new Map()` с устройствами из bridge/devices, но нет механизма очистки старых записей. Со временем может накапливаться.

### 6. TypeScript не компилируется — воркспейс не настроен [LOW]
Попытка `npx tsc --noEmit` показывает ошибки конфигурации TypeScript.
